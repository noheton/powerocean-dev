2. CHARGER-LEVEL WRITE  (the call that actually changes runtime behaviour)
================================================================================

The HTTP POST above only updates the EcoFlow-side catalog. The actual handover
to a custom central system is a SECOND step: a proto-encoded MQTT/IoT command
sent over the same iot bus this integration already taps for sensors.

Wire-up:
  com/ecoflow/cp307module/util/o.java:916
  ─────────────────────────────────────────
  public final void X(String deviceId, String url, String vendor,
                      int profile, String authKey, dd.g cb) {
    Cp307Ocpp.vendorInfoSet msg = Cp307Ocpp.vendorInfoSet.newBuilder()
        .setDeviceId(deviceId)   // OCPP charge-point ID
        .setUrl(url)             // ws:// or wss://
        .setVendor(vendor)       // CPOcppBindReq.platformCode
        .setProfile(profile)     // OCPP 1.6 security profile: 0/1/2
        .setAuthKey(authKey)
        .build();
    C(this, 53, 224, 161, 0, 0, msg, 0L, new y(), 88, null);
    //          ^module ^cmd_set ^cmd_id (VENDOR_INFO_SET = 0xA1)
  }

4. RESET / FALLBACK PATH  — open question RESOLVED
================================================================================

doc/apk.md:214 listed this as unresolved: "Is there a 'reset to EcoFlow cloud'
call, or does the user need a factory reset?"

Answer: YES — proto cmd VENDOR_INFO_CLR (0xA3) clears the runtime config on
the charger. It is independent of the HTTP catalog. To fully revert:

  1. Send VENDOR_INFO_CLR over the iot bus  → charger reconnects to EcoFlow.
  2. Optionally POST /ocppPlatformConfig with isEnabled=0  → tidy the catalog.

No factory reset required. No HTTP DELETE exists.

Recipe used:
  grep -rn "resetOcpp|defaultOcpp|ecoflowOcpp|deleteOcpp|removeOcpp" sources/
  → 0 hits (the reset is a proto cmd, not a separate HTTP/Java method)

6. UPDATED HA INTEGRATION PLAN  (replaces apk.md Stage 1 sketch)
================================================================================

Two writes are needed for a working redirect — only one is enough to break
EcoFlow control. Order matters.

  Step 1 (HTTP, account-side, mostly cosmetic):
    POST /provider-service/app/ocppPlatformConfig
      body = CPOcppBindReq(
        platformCode="lbbrhzn",  platformName="HA lbbrhzn/ocpp",
        platformType=0,          // confirm enum from existing /list responses
        backendUrl="ws://homeassistant.local:9000",
        secureUrl="",            authKey="",
        sortOrder=0,             isEnabled=1,
        sn=<charger_sn>)
    → registers the platform in the user's account. Catalog only.

  Step 2 (PROTO, charger-side — this is the real switch):
    Send VENDOR_INFO_SET (CmdID 0xA1, module 53, cmd_set 224) carrying:
      device_id = <CP ID matching lbbrhzn/ocpp Charge Point Identity>
      url       = "ws://homeassistant.local:9000"
      vendor    = "lbbrhzn"
      cpo_name  = "HA lbbrhzn/ocpp"
      profile   = 0          // unauthenticated (default for lbbrhzn/ocpp)
      auth_key  = ""
    → the charger reconnects to the supplied URL on receipt.

  Step 3 — Reverse path:
    Send VENDOR_INFO_CLR (CmdID 0xA3, no body)  → reverts to EcoFlow cloud.

Read-only probes that need no decompile, only a bearer token:
    GET /iot-service/ac305/charge/ocpp/domain?sn=<sn>
    GET /provider-service/app/ocppPlatformConfig/list
    Send VENDOR_INFO_GET   (read what the charger currently believes)

8.3 Security profile  — `profile: uint32` in vendorInfoSet
──────────────────────────────────────────────────────────

  OCPP 1.6 Security Whitepaper Edition 2 defines:
    Profile 1: HTTP Basic auth over unsecured ws://
    Profile 2: HTTP Basic auth over wss:// (TLS)
    Profile 3: TLS + client cert authentication

  Profile 0 is NOT defined by the spec. It is the de-facto "no auth, plain
  ws://" mode that lbbrhzn/ocpp ships with by default; lbbrhzn/ocpp does
  not advertise a profile number for this mode in its README.
  We therefore read the EcoFlow proto's `profile=0` as "unauthenticated"
  and `profile=1` as the standard ws://+basic-auth mode that lbbrhzn/ocpp
  supports if the user sets a password.
  ★ Recommendation: when the writer is implemented, default profile=0 in
    the form (matches lbbrhzn/ocpp's out-of-box config); switch to
    profile=1 automatically when the user fills in `auth_key`.

8.4 lbbrhzn/ocpp host coordinates — what the user actually runs
───────────────────────────────────────────────────────────────

  Confirmed from custom_components/ocpp/const.py:
    DEFAULT_PORT = 9000
    CONF_CPID    = "charge_point_identity"

  This validates the form copy already in this branch:
    - Default URL hint  "ws://homeassistant.local:9000"
    - CP-ID label "must match the Charge Point Identity in lbbrhzn/ocpp"

  No path component is required — the charger connects to the bare host:port
  and lbbrhzn/ocpp accepts any path; the cpid sent in BootNotification.req
  is what binds the session to a configured charger entry.

8.5 OCPP-J URL format — chargePointIdentity in the path
────────────────────────────────────────────────────────

  The OCPP 1.6-J specification (OCPP-J 1.6 §3.1.1) defines the WebSocket
  URL as:
      ws://<host>[:<port>]/<centralSystemPath>/<chargePointIdentity>
  with the chargePointIdentity being the last URL segment.

  EcoFlow's vendorInfoSet packs this differently:
    url       = full base URL (no identity in path)
    device_id = the identity (sent separately, charger inserts on connect)

  This means the user enters only the BASE URL in our form. The integration
  must NOT append cpid to the URL — the firmware does that itself from
  device_id. Important UX note for the form description.
